This project aims to develop an algorithm for automatically generating logical axioms that can correctly describe the behaviour of DEVS models.

Below you can find a pseudocode description for the algorithm.
We show the algorithm divided in parts to make the description clear.

<img width="844" height="1034" alt="image" src="https://github.com/user-attachments/assets/bdaea893-9962-4a5d-8566-3777ef92a7df" />

The algorithm iterates through the DEVSMap file and converts each section into Axioms, accumulating them in a master list. Each Axiom contains a structured Formula that preserves operator precedence. As the algorithm reads a section, it identifies its type and invokes the corresponding generator: state variable sections use a type definition generator; input and output port sections share a common generator; and all remaining sections, which define variable assignments, use the general assignment axiom generator. Transition sections add an extra condition ensuring that only one transition is active; this condition is attached to every Axiom produced from that section. Each generator returns a set of Axioms, which are appended to the master list. After all sections are processed, the algorithm prints each Axiom by translating its Formula into the target language and writing it to the output file

The generate_state_variable_axioms function processes the JSON map of state variable facts by iterating through each variable–type pair and producing two Axioms per entry. For each variable, it builds a Formula of the form “name : type,” wrapped in a TPTP type Axiom so the variable is properly declared for automated theorem provers. All numeric types are normalized to real numbers to avoid TPTP type mismatch issues. The function then creates a second, identical Axiom for the corresponding next state variable, ensuring both current and next versions are consistently typed. After all entries are processed, the function returns the full list of generated Axioms.

<img width="851" height="854" alt="image" src="https://github.com/user-attachments/assets/76299e66-bd58-412f-bcb5-1abbdf286871" />

Then, the generate_port_axioms function processes a port definition map along with a label indicating whether the section defines input or output ports. It starts by creating an empty list of axioms and collecting all port names. It first introduces the port type itself by asserting that the given portType is a new, unrestricted TPTP type. The function then iterates through each port–message type pair. For every port, it records the name, creates an axiom declaring the port as an instance of the custom port type, and adds another axiom introducing a variable representing the messages sent or received on that port. After all ports are processed, the function generates two axioms quantified over the full list of port names: one stating that all ports of this type are distinct, and another stating that these ports are exactly the members of the custom port type. Finally, it defines a typed function symbol mapping any port to the number of associated messages, returning a natural number. The complete list of axioms is then returned.

The generate_variable_assignment_axioms function processes a Map whose keys represent state variable conditions. Each key maps either to another condition Map or to a Map of variable assignment expressions. As the function descends through the structure, it accumulates the conditions required to reach the current Map. When it reaches a Map whose values are no longer Maps, it has arrived at the base case: a set of variable assignments. It creates a Formula for each assignment, using the “next” version of the variable if it is a state variable. These assignment Formulas are combined with the accumulated condition Formulas into a single implication: if the conditions hold, then the variables take these values. This implication becomes an Axiom, which is added to the list of generated Axioms. The function returns all such Axioms produced during the recursive traversal.

<img width="841" height="734" alt="image" src="https://github.com/user-attachments/assets/dbb939bf-d068-4ce4-a97a-df053d1c954b" />

In the recursive case, the facts Map contains keys that are state variable conditions whose values are themselves Maps. The function iterates over every key except "Otherwise", converts each condition into a Formula, stores it in a list for later negation, and makes a recursive call using the Map associated with that key along with the accumulated conditions plus the current one. All returned Axioms are added to the main list. If an "Otherwise" branch exists, the function negates all previously collected condition Formulas—representing the case where none of the earlier conditions hold—and makes a recursive call using the "Otherwise" Map and this list of negated conditions. This produces Axioms stating what must happen when no other condition is true. The function then returns the full set of Axioms gathered during the traversal.
